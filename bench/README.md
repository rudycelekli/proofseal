# ProofKit Benchmark Suite (`bench/`)

The benchmark defined in **ADR-0001 §3** — the only place the ProofKit v0.1
headline claim is allowed to come from. The README claim is copy-pasted *from*
`bench/results/report.md`, never written first (Playbook rule 2b).

## Run it

```bash
node bench/run.mjs
```

Plain Node ESM, **zero npm dependencies** (`node:child_process` `execFile`,
`node:crypto`, `node:fs` only). No network access: the three fixture repos are
vendored and committed. Output: `bench/results/report.md` + `report.json`.

The suite runs even when ProofKit itself is not yet built: if
`dist/cli/index.js` is absent, the report is emitted marked **INCOMPLETE**
with the ProofKit column `pending (not built)`. Build ProofKit and re-run.

## What gets measured (ADR §3.2)

For ProofKit and each named incumbent baseline, on three vendored fixture
repos each carrying **exactly 100 claims**:

| Metric | Protocol |
|---|---|
| Setup steps (count) | discrete commands a maintainer runs, incl. key/OIDC steps |
| Setup time (median, s) | median of 3 scripted runs, fresh temp copy each |
| Config LOC | non-blank, non-comment lines the maintainer authors |
| Secrets to manage | count of secret artifacts to store/rotate |
| Verify latency p50/p95 (ms) | 20 cold-process verify runs per fixture |
| Tamper signaled (% of 45) | seeded fault-injection suite, seed 42 — any signal, tool's own semantics |
| Four-state classification (ProofSeal taxonomy) | ProofSeal-only score; competitors: N/A — taxonomy absent |
| Drift vs regression distinction | capability: yes / N/A — capability absent |
| Temporal history + bisection | capability: yes / N/A — capability absent |

## Baselines (ADR §3.1)

- **B1 `baselines/checksum/`** — the folk solution: `sha256sum -c SHA256SUMS`
  plus a hand-rolled grep loop over a marker list.
- **B2 `baselines/cosign/`** — sigstore cosign, both **keypair** mode
  (`generate-key-pair` / `sign-blob` / `verify-blob`, run hermetically with
  `--tlog-upload=false` / `--insecure-ignore-tlog` — no Rekor in the loop)
  and **keyless** mode. Keyless signing needs an OIDC identity (interactive
  browser flow or ambient CI token), which cannot run headless: the bench
  measures what is static about the procedure (setup steps, config LOC,
  secrets count — all properties of the workflow itself) and records every
  wall-clock/execution cell as **`N/A — requires interactive OIDC`**. That
  is itself an adoption-cost finding. Set `PROOFKIT_BENCH_COSIGN_KEYLESS=1`
  to opt in to the interactive flow and measure the rest.
- **B3 `baselines/intoto/`** — minimal in-toto layout (reference 1.x
  implementation): `in-toto-keygen` + signed layout + `in-toto-run`
  (products) + `in-toto-verify`. The layout carries an **inspection** with
  per-file `REQUIRE` → `MATCH` → `DISALLOW` rules: `in-toto-verify` alone
  only checks recorded link metadata, so verifying the *live tree* (the job
  this bench measures) requires re-recording the cwd at verify time — that
  inspection cost is part of in-toto's honest latency number.

Tool availability is detected at runtime. An absent tool produces an explicit
**`SKIPPED (not installed)`** cell in the report — never a silent omission. A
tool that is installed but whose ceremony cannot run headless (cosign keyless)
gets a **static-only** column: measured setup-cost cells, `N/A — <reason>`
execution cells.

## Fixtures (ADR §3.3)

Generated by `node bench/fixtures/generate.mjs` (seeded LCG, idempotent —
re-running produces byte-identical output) and **committed**; the bench never
fetches anything. Each fixture carries `claims.json` with exactly 100 claims,
and `run.mjs` self-checks fixture integrity (file hashes + harness expected
outputs) before benchmarking.

| Fixture | Files | Claim mix |
|---|---|---|
| `repo-a-npm-lib` | ~35 (TS lib + committed deterministic `dist/`) | 20 file-hash, 80 marker |
| `repo-b-python-tool` | 20 (Python CLI + deterministic harness) | 5 file-hash, 93 marker, 2 harness |
| `repo-c-docs-site` | 15 (Markdown/config) | 5 file-hash, 95 marker |

Claim-allocation rule: marker-host files carry **only** marker claims;
file-hash claims cover only non-host files (`dist/`, docs without markers,
metadata). A pure file-hash claim regresses on any edit (it has no marker to
survive through), so "edit with marker intact → drift" is only a structurally
valid expectation when the edited file's claims are all marker claims.

The repo-b harness (`tools/deterministic.py`) uses an **inline Park–Miller
LCG** (not Python's `random` module), seeded from `PROOFKIT_SEED`, printing 16
floats. Integer state stays below 2^31, so `state / M` is an exact IEEE-754
double — byte-identical in CPython and JS, which is how the generator computes
the committed `expectedSha256` (quantize 6 decimals → LE f64 pack → sha256,
the ADR D9 pipeline) without requiring Python at generation time.

## Mutation suite (sub-claim C4)

`mutations/mutations.json`: **45 seeded mutations** (seed 42, ≥ 40 required)
across 4 fault classes, applied each to a fresh copy of the sealed tree:

| Class | Operation | ProofSeal expected class |
|---|---|---|
| `manifest-byte` (12) | single-byte mutation of the trust artifact (signature hex, claim sha256, summary count, manifest hash) | `fail` |
| `marker-removal` (12) | remove a claimed marker line | `fail` (regressed) |
| `edit-marker-intact` (12) | edit a claimed file, marker preserved | `drift` |
| `file-deletion` (9) | delete a claimed file | `missing` |

**Scoring (premortem round 3).** The only cross-tool score is **tamper
signaled**: did the tool raise *any* signal on the mutation, judged under the
tool's OWN semantics. A cosign or checksum hard FAIL on a benign append is
CORRECT for a byte-integrity model and counts as signaled — never
"misclassified". The four-state classification (pass / drift / regressed /
missing) is ProofSeal's OWN taxonomy: it is scored for ProofSeal only and
reads **`N/A — taxonomy absent`** for every other tool — a percentage on a
rubric only ProofSeal subscribes to would be a self-serving metric.

The checksum baseline is practitioner-grade: `SHA256SUMS` is generated over
ALL git-tracked fixture files (`git ls-files` at the seal point), not just
claim-referenced files.

## Honest-comparison policy (ADR D12)

- Where a baseline structurally cannot express a capability (B1 has no
  tamper-evident manifest; B2/B3 have no drift semantics, history, or
  bisection), the table records **`N/A — capability absent`** — that is itself
  a benchmark result, not a gap to hide.
- cosign is a baseline for **setup cost**, never framed as an inferior
  competitor on security: it answers *who signed this artifact*; ProofKit
  answers *are this repo's claims still true* (commit-bound tamper-evidence,
  not third-party authentication — ADR R1).
- The checksum baseline's `SHA256SUMS` is not tamper-evident; for B1 the
  manifest-byte class measures accidental corruption only. The report says so.

## Environment-dependent results

Reproducible everywhere: claim workloads, mutation list, harness expected
hashes, capability cells, mutation classifications. **Environment-dependent**:
wall-clock numbers (setup time, verify p50/p95 — machine/CPU/disk dependent),
tool availability cells (cosign / in-toto installed or not), and tool
versions. The provenance block records git SHA, seeds, OS, Node/Python/tool
versions, and total wall-clock for every run; compare latency numbers only
across runs from the same provenance environment.

## Layout

```
bench/
  run.mjs                  # orchestrator — node bench/run.mjs
  fixtures/
    generate.mjs           # seeded generator (committed output below)
    repo-a-npm-lib/        # fixture + claims.json (100 claims)
    repo-b-python-tool/    # fixture + claims.json (100 claims, 2 harness)
    repo-c-docs-site/      # fixture + claims.json (100 claims)
  baselines/
    checksum/baseline.mjs  # B1
    cosign/baseline.mjs    # B2 (keypair + keyless variants)
    intoto/baseline.mjs    # B3
  mutations/mutations.json # 45 seeded mutations (seed 42)
  lib/                     # zero-dep helpers + ProofKit CLI adapter
  results/                 # report.md + report.json (emitted)
```
